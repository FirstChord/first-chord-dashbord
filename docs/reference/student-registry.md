# Student registry contract

`lib/config/students-registry.js` is the hand-maintained source for student
portal configuration. It is not the source of lesson, calendar, billing, or live
tutor-assignment truth; MMS owns those facts.

## Normal student lifecycle

Use `/admin/onboard` to add a student and the student archive workflow to remove
one from active operation. Those workflows coordinate Sheets, the registry, MMS,
generated portal files, and recovery detail. A manual registry edit is a repair
or portal-configuration change, not a substitute for onboarding or archiving.

Onboarding can partially succeed. Check its recorded step results before retrying
anything; do not repeat the whole workflow merely because one downstream step
failed. See the **Waiting and onboarding** row in [AGENTS.md](../../AGENTS.md).

## Source and generated outputs

Edit only:

```text
lib/config/students-registry.js
```

Regenerate these outputs; never repair them directly:

```text
lib/student-url-mappings.js
lib/student-helpers.js
lib/soundslice-mappings.js
lib/config/theta-credentials.js
lib/config/instruments.js
```

Run:

```bash
npm run generate-configs
npm run validate
```

`npm run build`, `npm run dev`, `npm run test:admin`, and `npm run validate`
also regenerate configurations first. Always inspect the resulting diff and do
not commit local credential output or unrelated generated churn.

## Registry entry

```js
'sdt_XXXXXX': {
  firstName: 'Alex',
  lastName: 'Example',
  friendlyUrl: 'alex-e',
  tutor: 'Finn',
  instrument: 'Guitar',
  soundsliceUrl: 'https://www.soundslice.com/courses/12345/',
  thetaUsername: 'alexexamplefc',
  fcStudentId: 'fc_std_1234abcd',
}, // Alex Example
```

| Field | Meaning |
|---|---|
| Registry key | MMS student ID and cross-system join key |
| `firstName`, `lastName` | Portal display/configuration names |
| `friendlyUrl` | Unique public portal slug |
| `tutor` | Portal/config grouping; not live tutor truth |
| `instrument` | Portal/song-shelf override using the repository vocabulary |
| `soundsliceUrl` | Existing per-student Soundslice course URL |
| `thetaUsername` | Optional Theta portal credential source |
| `fcStudentId` | Stable First Chord identity generated by onboarding |

Escape apostrophes and backslashes using the same rules as
`lib/admin/registry-helpers.mjs`; do not hand-invent an FC ID or friendly URL when
the onboarding helpers can generate them.

For an existing student, edit **Instrument** on `/admin/students/[mmsId]`. That
single save deliberately aligns the Students sheet and this registry field. Use
the exact `Electric Guitar` option for the electric song catalogue; `Guitar`
selects the acoustic catalogue. The production registry commit triggers the
normal deployment path, whose prebuild regenerates `lib/config/instruments.js`.

## Tutor changes

Changing `tutor` in the registry does not move lessons or alter the live tutor
dashboard roster. Make the actual assignment change in MMS first, then align the
registry grouping if the portal configuration should follow it. For a genuine
dual-tutor relationship, follow the current school policy rather than assuming a
shared registry record is harmless.

## Safe repair sequence

1. Confirm the student and intended tutor in MMS and the `Students` sheet.
2. Change the canonical registry entry only.
3. Run `npm run generate-configs` and `npm run validate`.
4. Inspect all generated diffs, especially friendly URLs and tutor groupings.
5. Run the focused tests/build appropriate to the change.

For validation output and limitations, use
[Student registry validation](./student-registry-validation.md).
