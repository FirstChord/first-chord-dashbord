# External Audit — 2026-06-10

Independent critique of the dashboard, architecture, documentation, agent readiness, and forward plan. Produced by a review agent on 2026-06-10; reviewed docs (repo + Obsidian vault), loop/issue architecture, overview and issue-queue code, the sheets layer, auth, and ran the admin test suite (160/160 passing at time of audit).

This is critique, not instruction. Where it conflicts with `CURRENT_STATUS.md`, treat `CURRENT_STATUS.md` as the active direction and this file as input to it.

---

## Executive summary

This is an unusually disciplined internal-tools project. The loop pattern (Detected → Guided → Actioned → Logged → Resolved/Kept Active) is genuinely implemented, not just documented: `Issue_Queue` merge logic handles reappearance and reopen correctly, consequential actions write to `Event_Log`, vendor reads are cached or manual, and the test suite covers the helper logic that matters. Source-of-truth boundaries are written down *and* mostly respected in code — that's rare.

The main risks are not architectural. They are:

1. **Documentation sprawl** — at least five orientation entry points across three locations; the duplication will drift.
2. **State-tab sprawl** — ~12 dashboard-owned Sheets tabs, each with its own hand-rolled read-modify-write upsert in a 1,833-line `sheets.js`, with no locking and no shared write primitive.
3. **Concrete hygiene problems** (home-directory git repo, credentials in git, test data in production) that are cheap to fix and worth fixing before any "case study" framing.
4. Early signs of **overview-page noise creep**.

The "do not build yet" lists are correct. The pushback runs the other direction: a few items on the "next slices" list (communication draft layer, forecasting) are still premature, and one foundational item (Stripe snapshot caching) keeps being recommended in docs but never built.

## 1. Strengths

**The loop discipline is real.** `mergeIssuesWithQueueState()` (`lib/admin/issue-queue.js`) correctly handles the hard cases: source-managed issues going quiet (system-cleared), resolved issues still detected (reopen with event), and stable issue identity via `buildIssueId`/`contextKey`. This is the part most internal dashboards get wrong, and it's tested.

**Source-of-truth lanes are explicit and consistent.** The same ownership table appears in CURRENT_STATUS, NEW_AGENT_START_HERE, the blueprint, and the Obsidian vault, and the code matches it: the dashboard never writes MMS/Stripe truth except through narrow, audited paths (onboarding, archive, the pause-planning `payment_expectation` guardrail). The "transitional split ownership" label in OWNERSHIP_MATRIX.md is honest engineering — naming the compromise instead of pretending it doesn't exist.

**Approval-first is enforced structurally, not just by policy.** WhatsApp text is copy-only; the pause action requires a logged confirmation checkbox before the button exists; `Done` on a planning item deliberately doesn't mutate payment state. These are the right boundaries placed in the right layer (the action simply isn't wired, rather than "the agent is told not to").

**Test coverage is aimed at the right layer.** 160 passing tests over the `.mjs` helper modules — issue classification, queue merge, pause expectation, Stripe snapshot derivation. Pure-logic helpers separated from I/O (`*-helpers.mjs` vs `*.js`) is exactly the structure that makes future AI assistance safe.

**The docs are current.** CURRENT_STATUS and NEW_AGENT_START_HERE matched the actual code and recent commits on the audit date. That almost never happens.

## 2. Risks / weak spots

### Hygiene (fix first — cheap, and they undermine everything else)

- **The home directory (`/Users/finnlemarinel`) is itself a git clone of the dashboard repo.** Its `origin` is `FirstChord/first-chord-dashbord` and it tracks an old `music-school-dashboard/` copy (now mass deletions), plus a personal `study/` folder, `Desktop/Tools:Games/FC Admin Tools/`, and modified copies of dashboard config files under `Desktop/FirstChord/`. Anything committed and pushed from there lands in the dashboard's GitHub repo. It should be dismantled (or at minimum have its remote removed) deliberately.
- **Student credentials are tracked in git.** `lib/config/theta-credentials.js` maps student IDs to Theta usernames, and password equals username. This is in the GitHub repo. Low-stakes accounts, but still children's login credentials in a remote repo — and exactly the kind of thing that blocks a public case study later. It should move to an untracked/env-loaded artifact.
- **Test data lives in production data, filtered by hardcoded names.** `OVERVIEW_TEST_STUDENT_NAMES` in `app/admin/page.js` filters "test studenty" and "finn le marinel" out of overview counts, and commit `0b68af5` added a Test Student "[TEST — will remove]". Name-based filtering is fragile (it only applies on Overview, not `/admin/students`, flags counts, or capacity), and the removal promise lives only in a commit message.

### Architecture

- **`sheets.js` is becoming the god module.** 1,833 lines, with the same read-all-rows → find-by-id → append-or-update upsert hand-written per tab (`upsertShowcaseTaskStateRow`, `upsertScheduleContextRow`, `upsertParentUnderstandingStateRow`, `upsertHolidayWorkflowStateRow`, `upsertTutorAbsenceStateRow`, …). A shared *tab accessor* (header-mapped read, keyed upsert, append-log) is not a workflow engine — it's deduplication. The current pattern means every new workflow adds ~80 lines of copy-paste with its own subtle bugs.
- **No concurrency story for Sheets writes.** Every upsert is read-modify-write by row index with no lock or version check. With one admin this is fine; with Finn + Tom + Fenella using Planning and Parent Understanding simultaneously, two saves can interleave and one silently overwrites or lands on the wrong row (row indexes shift after an append). No database needed — but the limit should be known and documented, and keyed rows possibly re-read immediately before update. This matters more as delegation increases, which is the stated direction.
- **State-tab sprawl is the quiet complexity.** Dashboard-owned tabs now include Issue_Queue, Event_Log, Waiting_List_State, Showcase_Task_State, Holiday_Workflow_State, Tutor_Absence_State, Schedule_Context, Parent_Understanding_State, Planning_Items, Planning_Progress_Log, Students_Archive. Each is individually justified, but the *aggregate* is a schema nobody can see in one place. V3's own warning — "the risk is allowing each new loop to invent its own state, action, and audit semantics" — is slowly coming true at the storage layer even while the conceptual layer stays clean.
- **Client components are getting heavy.** `AdminIssuesPageClient` (~1,426 lines), `AdminPlanningPageClient` (~1,337), `AdminParentUnderstandingPageClient` (~1,100). Single-file form/state monoliths; the least-tested layer (tests cover helpers, not these).
- **Naming drift: flags vs issues vs Review_Flags.** The route is `/admin/flags`, the concept is "issues," `Review_Flags` is a brain-generated source tab, and the Overview counts `getReviewFlagsRows()` as "Open issues" — actually only one source of issues, not the merged queue. Worth one glossary paragraph and a decision to stop using one of the words.

### Dashboard clarity

- **Overview is starting to re-accumulate noise.** It now computes attention items, work-queue items, dedupe between the two, a priority sentence, a trust line, lifecycle pills, and two collapsed sections. Each addition was reasonable; the sum approaches the "sitemap of numbers" the nav redesign was meant to avoid. The tell: `buildPrioritySentence` truncates to two priorities because there are routinely more than two.
- **Old portal docs coexist with admin docs.** `docs/` root still carries student-portal generation docs alongside `docs/admin/`. Nothing tells a newcomer which half of `docs/` matters for which job.

## 3. Documentation gaps and duplication

**Could a new agent continue safely?** Yes — genuinely. CURRENT_STATUS + V3_LOOP_ARCHITECTURE + OWNERSHIP_MATRIX + the test suite is enough to extend the system without breaking its semantics. This is the strongest part of the project.

**But the entry-point count is now a liability.** A new agent can be pointed at: workspace `CURRENT_HANDOVER.md`, workspace `CLAUDE.md`, repo `NEW_AGENT_START_HERE.md`, repo `CURRENT_STATUS.md`, repo `docs/admin/INDEX.md`, `.claude/QUICK_START.md`, Obsidian `00 Home`, Obsidian `08 Operations/New Agent Start Here`, Obsidian `Current System Map`. Several restate the same source-of-truth table, recent-work list, and avoid-list. They agree today because they were updated together; the moment one update is missed, an agent will follow the stale one with confidence. Recommendation: pick **one** canonical orientation doc per audience (repo: NEW_AGENT_START_HERE; vault: Home) and make every other file a two-line pointer rather than a summary.

**Duplicated learning logs.** `docs/LEARNING_LOG.md` and Obsidian `06 Learning Log/` have overlapping mandates. Decide: repo log = engineering decisions tied to files; vault log = operating/product lessons — and have entries live in exactly one.

**Missing docs:**

- **A state-tab schema page.** One table: every dashboard-owned Sheets tab, its key column, owner, writer functions, and whether it's append-only. The single most useful missing doc for both new agents and future productisation.
- **Concurrency/limits note.** "Sheets writes are last-write-wins read-modify-write; here's what that means when two admins work at once."
- **A "who uses what" map.** Finn vs Tom vs Fenella per surface. Delegation is the stated goal; the docs describe the system, not its operators.
- **Secrets/credentials inventory.** What's in `.env`, what's tracked, what would need rotating if the repo leaked.

**As a case-study foundation:** strong. The blueprint doc is already 80% of a publishable article, and the learning log gives it dated receipts. The blockers are the hygiene items (credentials in git) and that the blueprint mixes the reusable pattern with First Chord vocabulary — fine for now, don't polish it yet.

## 4. Agent/automation readiness

**Structurally well-prepared, for the right reasons.** The things that make AI assistance safe already exist as side effects of good engineering: pure tested helpers separated from I/O, normalised state vocabularies, append-only event history, derived context with reasons/confidence/warnings, and approval gates that are structural (unwired) rather than policy ("don't").

**Approval boundaries:** clear for the two dangerous categories. Parent-facing = copy/paste only; payment-affecting = explicit button behind a logged confirmation, writing through the one existing PATCH route. One gap: the boundaries are documented per-feature, not as a single policy. Before any AI touches the system, write one short "automation boundary" doc listing the actions an agent may *never* perform and the routes that enforce that — so it can be handed to an agent as a contract instead of inferred from five docs.

**What should remain deterministic:** issue detection, queue merge/reopen semantics, payment/pause expectation rules, lifecycle derivation, capacity matching, and everything in `Event_Log`. AI should never be the reason an issue exists or clears.

**Safest first AI uses** (all read-only, all checkable against ground truth):

1. **MMS sign-up note parsing on `/admin/waiting`** — instruments are already parsed deterministically; an LLM extracting instrument/availability/age into the same structured hint format, displayed as a suggestion with the raw note alongside, is low-risk and immediately useful.
2. **Event_Log summarisation** — "what happened with this student" on the detail page, derived from existing audit rows. Wrong summaries are annoying, not dangerous, and the log is right there to verify.
3. **Planning-item triage suggestions** — suggested area/owner/next-action phrasing for inbox captures, accepted by click.
4. *Later, not now:* message drafting — only after the draft/approval record layer exists, because drafts without approval state become "communication history" by accident.

## 5. Generalisability / productisation

**First Chord-specific:** the vendor trio (MMS, Stripe restricted key, the specific Sheets schema), the registry/generate-configs pipeline, `first-chord-brain` identity generation, school policy, and the workflow definitions (showcase, holidays, parent understanding).

**Genuinely generalisable patterns** (the real asset):

- The loop taxonomy itself — issue loops vs workflow loops vs queue loops vs vendor-truth loops (`V3_LOOP_ARCHITECTURE.md` is the best doc in the repo).
- Issue queue with source-presence semantics (system-cleared, reappear, reopen-if-still-detected).
- The ownership matrix format, including "transitional split ownership" as an honest state.
- Approval-first action design and the automation escalation ladder.
- Cached vendor reads with explicit refresh and visible freshness.
- Action-led navigation (Overview/Issues/Workflows/Planning).

**What would need abstracting before productisation:** the storage layer (Sheets-tab-per-workflow with hand-rolled upserts is the least portable part), issue *detectors* as pluggable rules vs the queue *engine* (the engine is nearly generic already; the detectors are 100% First Chord), and a real tenant/config model. That's rewrite-scale, which is why:

**Don't productise yet — the right next step toward productisation is *writing*, not extracting code.** The blueprint + learning log as a case study has near-term value (reputation, clarifying the model, attracting the "other music school" conversation that would reveal what to abstract). Extracting a product before a second real user exists would freeze today's abstractions, several of which (Sheets storage, single-admin concurrency) are already known to be temporary.

## 6. Recommended next slices

The CURRENT_STATUS "best next slices" list is mostly sound but it's a 10-item menu, which dilutes the slice discipline. Top three:

**Slice 1 — Hygiene and trust floor (smallest, do first).** Dismantle the home-directory git repo (remove its remote at minimum, after checking nothing unpushed matters). Get `theta-credentials.js` out of git tracking (generate to an ignored path; rotation likely overkill given the threat model). Remove the test student per the commit's own promise, and delete the hardcoded name filter or replace it with a registry/sheet flag that all surfaces respect.

**Slice 2 — State-layer consolidation, not expansion.** Before the next workflow tab is added: one shared keyed-upsert/read helper in `sheets.js` that the existing tabs migrate to incrementally, plus the missing **state-tab schema doc**, plus a one-paragraph concurrency note. This is the "boring" slice that keeps the next six months cheap, and it directly serves delegation.

**Slice 3 — Pause loop maturity (the docs' own repeated recommendation — finish it).** `V3_LOOP_ARCHITECTURE.md` has recommended "make pause issues state their evidence source (Pause History vs sheet expectation vs live Stripe)" for several iterations; it keeps being deferred in favour of new surfaces. Closing it would make pause the second fully-mature loop and a template for the rest. Bundle the also-repeatedly-recommended **cached Stripe snapshot** here only if pause-issue clarity genuinely needs it; otherwise leave Stripe manual.

**Pushback on the current list:**

- **Communication draft layer (CURRENT_STATUS item 8): still premature.** The contact role model (item 6) is its prerequisite and is listed *after* it. Sequence: contact roles → draft/approval records → (much later) sending.
- **Forecasting/planning: not yet.** Payment value context is one slice old and self-described as "baseline, not accounting." Forecasting built on an unhardened value layer will produce confident wrong numbers — worse than no numbers. Let value context survive a term of real use first.
- **Watch the workflow-per-month pace.** Parent understanding, planning inbox, tutor absence, archive, and student-linked planning all landed within weeks. Each is individually disciplined, but new *surfaces* are outpacing the maturation of existing loops (pause; onboarding step-tracking — flagged in OWNERSHIP_MATRIX as the "highest-value hardening target" and still not done). The next quarter should mature more than it adds.

## Things not to build yet (confirming and extending existing lists)

The existing "do not" lists are correct: no WhatsApp auto-send, no Stripe mutations, no lifecycle state machine, no workflow engine, no new database, no heavy ownership system. Additions:

- **No AI drafting of parent messages** until draft/approval records exist (the existing gate covers *sending*; drafting needs its own).
- **No productisation/extraction** — write the case study instead.
- **No forecasting layer** until payment value context has survived real use.
- **No third orientation doc.** The next documentation work should delete/merge, not add.

The system is doing the rare thing it set out to do: staying boring on purpose. The biggest threat isn't a bad architectural decision — it's accumulation: of docs, of tabs, of surfaces, of overview cards. The principle that got it here ("clarity and closed loops before complexity") now needs applying to the system's own artifacts, not just its features.
